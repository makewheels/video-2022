| 功能 | 模块 | 说明 |

| 应用版本检查 | 后端 / Android / iOS | `checkUpdate` 已接通 |

### ⚠️ 仍需完善

| 方向 | 状态 | 说明 |
|------|------|------|
| 控制台自动化测试 | 较弱 | `console/` 目前没有成体系的单元测试或页面测试 |
| 播放链路 E2E | 中等 | 单元测试已覆盖，但浏览器级端到端还没有验证完整播放会话闭环 |
| 生产环境安全配置 | 需运维配合 | `FILE_ACCESS_SIGNATURE_SECRET` 在生产环境必须显式配置 |
| 文档历史包袱 | 持续治理中 | `docs/plans/` 和部分归档文档保留旧目录名，仅作历史参考 |

## 二、代码质量观察

### 最近完成的收口

- 播放器已经切到 `video.js`，播放会话协议与后端 DTO 对齐
- `/file/access` 现在会校验签名和时间戳，不再是“带 sign 参数但不验证”
- 全局异常不再把完整 stack trace 暴露给客户端
- 控制台不再并行维护两套开发者应用入口，统计与 Webhook 页面已接到真实后端

### 当前仍值得继续盯的点

- `console/` 缺少测试，后续字段漂移更难第一时间发现
- 播放会话与旧版 `Heartbeat` 目前双轨并行，后续是否合并需要明确策略
- Python E2E 主要覆盖页面骨架和接口可用性，复杂交互仍以单元测试为主

## 四、改进建议

### 短期

1. 为 `console/` 增加 API client 和核心页面测试，先覆盖 Stats / Webhooks / Apps
2. 增加浏览器级播放 E2E，验证 `start → heartbeat → exit` 的真实链路
3. 在部署检查里强制校验 `FILE_ACCESS_SIGNATURE_SECRET` 是否已配置

### 中期

4. 明确播放统计双轨策略：保留 `Heartbeat` 与 `PlaybackSession`，还是逐步收敛到单一模型
5. 为 CLI 增加面向真实后端的集成测试，而不只是 mock + 少量 smoke case
6. 继续刷新历史文档，把常用入口与历史设计稿分层得更清楚
